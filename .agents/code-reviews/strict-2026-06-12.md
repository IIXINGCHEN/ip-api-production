### BLOCKER 3 — Two parallel rate limiters gate the same requests

```
severity: blocker
file: src/app.js + src/middleware/rateLimitFixed.js + src/routes/geo-modern.js
symbol: createRateLimitMiddleware (app.js line 280) and rateLimitMiddleware (geo-modern.js line 185)
issue: independent in-memory limiters both count the same request
detail: app.js mounts rateLimitFixed.js (max=100/15min, Map<ip, RateLimitEntry>)
        globally. Inside geo-modern.js, the per-route rateLimitMiddleware uses
        SecureCache as a Map<ip, timestamps[]> with a separate 100/min budget.
        Effective limit is whichever trips first, but the two stores double
        memory and report conflicting metrics. Worse, the per-route store is

suggestion: Pick one. Recommended: drop the per-route rateLimitMiddleware

        single source of truth. Also bound the rate-limit map size and rely
        on Cloudflare's per-isolate lifecycle to reset state.

### MAJOR 2 — IPv6 validation delegates to URL constructor; false negatives

```
severity: major
file: src/utils/ipValidation.js
symbol: isValidIPv6 (line 18–29)
issue: uses new URL(`http://[${ip}]/`) to validate IPv6
detail: The browser URL parser accepts many strings URL would parse but that
        are not valid IPv6 (it tolerates zone IDs that some platforms do not,
        and rejects some legal compressed forms depending on runtime). It is
        also stricter than zod's z.string().ip() in some V8 versions and
        looser in others. This produces flaky validation that varies by
        runtime. The ipParamSchema in geo-modern.js already uses zod's
        z.string().ip() before this validator is called, so the two diverge
        silently.
suggestion: Replace isValidIPv6 with a regex / manual octet parser. The
        simplest correct implementation:
          const re = /^(?:[0-9a-fA-F]{1,4}:){7}[0-9a-fA-F]{1,4}$|^::1?$|^(?:[0-9a-fA-F]{1,4}:){1,7}:$|^(?:[0-9a-fA-F]{1,4}:){1,6}(?::[0-9a-fA-F]{1,4}){1,6}$/;
        Or use net.BlockList in Node test envs, but rely on zod for the
        request-facing validation since it's already wired.
```

symbol: isLegitimateISP / isLegitimateService / isLegitimateDatacenter
issue: extremely brittle single-IP gating for "legitimate" decisions
detail: threatService.checkVPN (line 163) short-circuits to "not a VPN" the
        moment isLegitimateISP(ip) is true. The whitelist file is a 541-line
        hand-curated list of single IPv4 addresses. Any IP outside the list
        (including real Comcast/Verizon/Charter customers) gets scored as
        a VPN risk. This produces both false positives (legit users

        gate for VPN detection.
suggestion: Use CIDR ranges (e.g., net.BlockList in Node) for ISP networks
        rather than individual addresses; better, replace isLegitimate* with
        a default-deny and use the whitelist only for known-good infrastructure
        (Cloudflare, Google, AWS ranges). Mark this as a known limitation
        until then.
```

### MAJOR 4 — sensitivePatterns in secureLogger masks IPs in any string > 20 chars

```
severity: major
file: src/utils/secureLogger.js (line 13–14, 50–62)
symbol: secureLogger.sanitizeMessage
issue: over-eager redaction silently corrupts logs
detail: pattern /[a-zA-Z0-9]{20,}/g replaces any long alphanumeric run with
        "***" or maskedKey. This catches:
          - cache keys (data.compress generates long base64 strings)
          - hex-encoded SHA digests
          - UUIDs (request IDs)
          - base64-encoded user agents
        The output log then no longer matches what the caller logged, and
        the maskKey helper shortens to first-6 + "***" + last-4, which leaks
        both prefix and suffix — defeating the purpose of the redaction for
        any string with high entropy.
suggestion: Restrict the long-string mask to values explicitly tagged as
        sensitive (apiKey, token, secret). Replace the global long-string
        rule with a per-key allowlist approach already used by
        sanitizeObject's sensitiveKeys logic.
```

### MAJOR 5 — SecureCache "encryption" is reversible XOR with a key derived from the cache key

```
severity: major
file: src/utils/secureCache.js
symbol: SecureCache.encrypt / generateEncryptionKey
issue: ciphertext is recoverable by any caller with the same key derivation
detail: encryption key = (cacheKey + ENVIRONMENT.current + 'secure-cache-key')
        .substring(0, 32). Anyone with the original cache key (which the
        caller already knows — they used it to set) can trivially decrypt.
        The `btoa(encrypted)` round-trip also has known footguns for
        non-ASCII content (the String.fromCharCode(...uint8) approach breaks
        on values > 255 when using TextEncoder/Decoder). Production traffic
        could see decryption errors that the code swallows and returns null
        for, masking the real cause.
suggestion: Either drop the encryption wrapper (call it obfuscation) or use
        WebCrypto AES-GCM with a stable environment-scoped key. At minimum,
        wrap the btoa round-trip in a try/catch and surface the failure
        via stats.
```

### MAJOR 8 — `app.get('/:ip{[0-9A-Fa-f:.]+}')` is in the file but not exported in the live handler chain

```
severity: major
file: src/routes/geo-modern.js (line 435) vs src/app.js (line 777)
symbol: app.get('/:ip...') (in geo-modern.js) is shadowed by app.get('/{ip}', c.json(...)) in app.js
detail: app.js mounts geoRoutes on '/', but app.js also defines its own
        app.get('/', ...) handler which does NOT forward to geoRoutes.
        The intended routing is: root handler decides whether to delegate
        to the route module's /:ip subpath. Because Hono registers handlers
        per-prefix, and geoRoutes is mounted at '/', a request to /1.2.3.4
        will try to match in order: app.get('/', ...), then geoRoutes'
        app.get('/:ip', ...). It will reach the route module's handler. But
        a request to "/" with `format=text` (handled in app.js line 301)
        returns text before geoRoutes' '/' handler runs. The two '/'
        handlers do not coexist gracefully — the result depends on
        registration order. The intent is documented, but easy to break
        during future maintenance.
suggestion: Refactor so the route module owns the '/' handler and app.js
        only does middleware setup. This eliminates the shadowing ambiguity.
```
